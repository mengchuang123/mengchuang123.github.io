RPC远程调用

简介

RPC是远程调用过程，是一种计算机网络协议，允许一个计算机的程序调用另外一个地址空间的函数。

RPC是一种典型的server/client形式，通过 发送请求和接受回应

远程函数调用与挑战

  • Call ID映射。我们怎么告诉远程机器我们要调用Multiply，而不是Add或者FooBar呢？在本地调用中，函数体是直接通过函数指针来指定的，我们调用Multiply，编译器就自动帮我们调用它相应的函数指针。但是在远程调用中，函数指针是不行的，因为两个进程的地址空间是完全不一样的。所以，在RPC中，所有的函数都必须有自己的一个ID。这个ID在所有进程中都是唯一确定的。客户端在做远程过程调用时，必须附上这个ID。然后我们还需要在客户端和服务端分别维护一个 {函数 <–> Call ID} 的对应表。两者的表不一定需要完全相同，但相同的函数对应的Call ID必须相同。当客户端需要进行远程调用时，它就查一下这个表，找出相应的Call ID，然后把它传给服务端，服务端也通过查表，来确定客户端需要调用的函数，然后执行相应函数的代码。

  • 序列化和反序列化。客户端怎么把参数值传给远程的函数呢？在本地调用中，我们只需要把参数压到栈里，然后让函数自己去栈里读就行。但是在远程过程调用时，客户端跟服务端是不同的进程，不能通过内存来传递参数。甚至有时候客户端和服务端使用的都不是同一种语言（比如服务端用C++，客户端用Java或者Python）。这时候就需要客户端把参数先转成一个字节流，传给服务端后，再把字节流转成自己能读取的格式。这个过程叫序列化和反序列化。同理，从服务端返回的值也需要序列化反序列化的过程。

  • 网络传输。远程调用往往用在网络上，客户端和服务端是通过网络连接的。所有的数据都需要通过网络传输，因此就需要有一个网络传输层。网络传输层需要把Call ID和序列化后的参数字节流传给服务端，然后再把序列化后的调用结果传回客户端。只要能完成这两者的，都可以作为传输层使用。因此，它所使用的协议其实是不限的，能完成传输就行。尽管大部分RPC框架都使用TCP协议，但其实UDP也可以，而gRPC干脆就用了HTTP2。Java的Netty也属于这层的东西。

MapReduce简单实现

总体框架

  • master-worker框架

1、存在进行任务分配，任务调度，差错管理的master进程和机器；

2、具体task由worker进行处理，worker进行循环getTask()使得分布式机器科研进行复用 思路借鉴与线程池的线程复用技术

3、通过远程调用分配任务，同时每一个worker程序完成任务以后，返回done信号，同时返回中间文件地址和命名信息

  • map-reduce框架

1、将大的处理任务和处理文件划分为M个split任务 交给worker处理

2、采用映射是思想，将每一个split文件中的ihash(key)%nReduce 划分为R个输出文件，使得不同小文件中同一个键值最后被同一个reduce worker处理

3、map操作按照键值进行快速排序，处理成为R个中间文件; reduce 操作进行归并排序

4、哈希算法：FNV哈希算法获取key值的32位的一个hash值

  • 差错处理

1、master每次分配一个任务以后，开启一个线程进行任务计时，如果超时没有受到Done信号，就重新将任务加入待分配的队列中，重新分配给其他机器进行处理

2、出错处理中的文件命名技巧，先保存在为系统的临时文件，临时文件是原子名字的，可以避免crash的机器文件对后续造成影响，最后处理完成再将临时文件保存为中间文件。

PTA 1021 Safe Fruit

There are a lot of tips telling us that some fruits must not be eaten with some other fruits, or we might get ourselves in serious trouble. For example, bananas can not be eaten with cantaloupe (哈密瓜), otherwise it will lead to kidney deficiency (肾虚).

Now you are given a long list of such tips, and a big basket of fruits. You are supposed to pick up those fruits so that it is safe to eat any of them.

Input Specification:

Each input file contains one test case. For each case, the first line gives two positive integers: N, the number of tips, and M, the number of fruits in the basket. both numbers are no more than 100.

Then two blocks follow. The first block contains N pairs of fruits which must not be eaten together, each pair occupies a line and there is no duplicated tips; and the second one contains M fruits together with their prices, again each pair in a line. To make it simple, each fruit is represented by a 3-digit ID number. A price is a positive integer which is no more than 1000. All the numbers in a line are separated by spaces.

Output Specification:

For each case, first print in a line the maximum number of safe fruits. Then in the next line list all the safe fruits, in increasing order of their ID’s. The ID’s must be separated by exactly one space, and there must be no extra space at the end of the line. Finally in the third line print the total price of the above fruits. Since there may be many different solutions, you are supposed to output the one with a maximum number of safe fruits. In case there is a tie, output the one with the lowest total price. It is guaranteed that such a solution is unique.

    887. 鸡蛋掉落

    给你 k 枚相同的鸡蛋，并可以使用一栋从第 1 层到第 n 层共有 n 层楼的建筑。

    已知存在楼层 f ，满足 0 <= f <= n ，任何从 高于 f 的楼层落下的鸡蛋都会碎，从 f 楼层或比它低的楼层落下的鸡蛋都不会破。

    每次操作，你可以取一枚没有碎的鸡蛋并把它从任一楼层 x 扔下（满足 1 <= x <= n）。如果鸡蛋碎了，你就不能再次使用它。如果某枚鸡蛋扔下后没有摔碎，则可以在之后的操作中 重复使用 这枚鸡蛋。

    请你计算并返回要确定 f 确切的值 的 最小操作次数 是多少？

    思路：

    • dp[k][N]代表目前有k个鸡蛋，等待确定的楼层有N层
    • dp[k][N] = min(dp[k][N],max(dp[k][N-x],dp[k-1][x-1])+1) 其中， (1<=x<=N-1)
    • 初始条件
      • k=1单独考虑
      • n=0||n=1单独考虑

    逆向思路：

    • f[t][k]我们可以进行t次操作，而且有k个鸡蛋，我们可以找到的最高的楼层n是多少

    • res的结果是找出最小的满足f[t][k]>=n的t即可

    • 地柜方程： f[t][k] = f[t-1][k] +f[t-1][k-1] +1

    进程调度

    概念

    CPU调度程序实现上下文切换，从就绪队列中选出一个进程到CPU中。

    • 就绪队列保存的是PCB
    • 就绪队列的组织方式：先进先出、优先队列、链表、树

    抢占调度：

    • 非抢占调度：调度仅仅发生在下面两种
      • 运行状态切换到等待状态（IO）
      • 进程终止
    • 抢占调度
      • 不是非强制

    非抢占下，一旦某个进程分配到CPU，改进程就回一直使用CPU,直到他终止或者切换到等待状态

    调度算法

    • 先到先服务
    • 最短作业优先 : 最短下次CPU执行时间
      • 对于批处理的长期调度，经常用于长期调度，但是无法再短期调度中实现，也无法获得下次CPU执行的长度，最多只能进行预测
      • 抢占式的最短作业优先调度又称为最短剩余时间优先
    • 优先级调度
      • 可能的问题：无穷阻塞或者饥饿，也就是优先级低的进程可能永远无法获得CPU时间
      • 解决方法：逐步增强等待很长时间的优先级
    • 轮转调度
      • 会定义一个较小的时间片，每次分配不会超过一个时间片的时间
      • 主要针对于分时系统，而且是抢占式的
      • 会经过多次的上下文切换，使得CPU是等待时间较长，也就是调度延迟很大
    • 多级队列调度

    线程调度

    在支持线程的操作系统上，内核级线程（不是进程）才是CPU调度的

    用户级线程由线程库管理，内核并不知道，用户线程运行在CPU上，最终映射到内核级线程

    用户级线程和内核级线程的映射可能不是直接的，可能会经过轻量级线程（LWP）

    多处理器调度

    目的：使得各个CPU负载分配平衡

    方法：

    • 非对称多处理：有一个主要用于调度的CPU，其他处理器仅仅执行代码

    • 对称多处理：所有处理器都自我调度（SMP）

    处理器亲和力

    • 大多数系统试图避免将进程从一个处理器转移到另外一个处理器上，而是试图让它固定在一个处理器上运行
    • 因为进程处理数据时处理器中的缓存不需要进行多次频繁的更新

    负载平衡：

    • 在SMP系统中所有CPU上的负载差不多
    • 方法：
      • 推迁移：一个特定任务检查负载
      • 拉迁移：空闲的处理器主动去拉任务
    • 负载平衡与处理器亲和度是矛盾的

    fork()和vfork()的区别

    fork（）与vfock（）都是创建一个进程，那他们有什么区别呢？总结有以下三点区别：

    1. fork （）：子进程拷贝父进程的数据段，代码段
      vfork （ ）：子进程与父进程共享数据段
    2. fork （）父子进程的执行次序不确定
      vfork 保证子进程先运行，在调用exec 或exit 之前与父进程数据是共享的,在它调用exec
      或exit 之后父进程才可能被调度运行。
    3. vfork （）保证子进程先运行，在她调用exec 或exit 之后父进程才可能被调度运行。如果在
      调用这两个函数之前子进程依赖于父进程的进一步动作，则会导致死锁。

    一致性： 实现其他三个特性的基础上，在应用层需要加判断的，用户恶意操作会导致不一致

    MVCC是什么 同时用到锁和log (回滚指针指向undo log)

    1、MVCC是什么

    InnoDB的多版本并发控制，优势是读不加锁，读写不冲突，读写分离，并发控制优势很大

    MVCC是指多版本并发控制。MVCC是在并发访问数据库时，通过对数据进行多版本控制，避免因写锁而导致读操作的堵塞，从而很好的优化并发堵塞问题。解决并发问题的通用方案有：

    （1）对并发访问的数据添加一把排它锁，添加锁之后，其他的读和写操作都需等待锁释放后才能访问。

    （2）添加一把共享锁，读读操作不需要等待锁的释放，读写和写写操作需要等待锁的释放。

    （3）通过对并发数据进行快照备份，从而达到无锁数据的并发访问。

    通俗的讲就是MVCC通过对数据进行多版本保存，根据比较版本号来控制数据是否展示，从而达到读取数据时无需加锁就可以实现事务的隔离性。

    2、解决什么问题

    • 脏读
    • 不可重复读
    • 幻读
    • 不能解决丢失更新的问题

    3、原理是什么

    MVCC实现由： 3个隐式字段，undo日志 ，Read View 三部分组成

    1、mvcc会在每一行后面加上隐藏字段，表示创建时间和删除时间，创建时间实际上是版本号，事务的id，回滚指针

    2、每一次执行一个新的事务版本号会增加

    3、undo log保证事务原子性

    4、读视图是用来保证可见性的，

    • 可重复读中，事务开始时创建一个视图，后面的操作都用一个视图，读到版本号等于当前版本号或者早于当前版本号的数据
    • 读提交时，每个语句执行都会创造一个视图，每次都是读到版本号最新的数据行

    4、使用的隔离级别

    仅仅在可重复读和提交读的隔离性下工作 都是通过版本号来操作的

    提交读 就是读取最新版本号的数据行

    可重复读： 查找版本号早于或者等于当前版本号的数据行

    在可重复读的基础上没有完全解决幻读的问题，还需要 RR上加上间隙锁

    所以引发了 当前读 和 快照读

    当前读 会加锁，保证读取的最新的版本

    快照读 性能考虑，但是读到的不一定是最新的 selete不加锁就是

    ArrayList和LinkedList的实现

    ArrayList和LinkedList都是实现List接口

    ArrayList底层是根据动态数组实现的，所以他添加删除的效率比较低，查找效率比较高

    LinkedList底层根据双向链表实现

    MyISAM和InnoDB的区别

    • InnoDB支持事务，MySIAM不支持事务
    • InnoDB支持行级锁
    • InnoDB支持外键索引
    • InnoDB不支持全文索引 FULLTEXT类型的索引
    • InnoDB没有区分索引文件和数据文件

    JUC线程池

    1、用来干嘛？

    采用池化的实现来管理过多的线程，可以提高乡响应速度等。主要的类有ThreadPoolExecutor

    2、运行机制？

    • 自身状态管理
    • 任务管理：充当生产者的作用，判断任务提交后是否申请线程或者放入任务缓冲队列。
    • 线程管理：分配的线程会进行任务的执行，执行完再去取新的任务，直到获取不到任务被回收

    自身状态管理

    同时维护两个变量

    • 运行状态：running shoutdown stop tidying terminiated 5个状态
    • 线程数量

    任务管理

    是线程池的主要入口，主要作用是

    • 检测当前线程池的状态
    • 后序步骤：直接分配线程，还是进入任务缓冲队列，还是直接拒接

    不同的任务缓冲队列会有不同的任务存取策略，

    线程管理

    为了实现线程管理，线程池内部设计Worker线程，实现AQS框架实现独占锁，线程管理的任务：

    • 长时间不运行的线程回收

    • 采用哈希表找到对线程的引用，通过哈希表实现对线程的管理

    • 调用runWorker()方法执行任务，runWorker（）循环通过getTask()方法获得任务

    任务执行

    线程池的工作线程通过Woker类实现，在ReentrantLock锁的保证下，把Woker实例插入到HashSet后，并启动Woker中的线程。 从Woker类的构造方法实现可以发现: 线程工厂在创建线程thread时，将Woker实例本身this作为参数传入，当执行start方法启动线程thread时，本质是执行了Worker的runWorker方法。 firstTask执行完成之后，通过getTask方法从阻塞队列中获取等待的任务，如果队列中没有任务，getTask方法会被阻塞并挂起，不会占用cpu资源；

    操作系统结构

    内核结构

    使用系统调用提供服务

    微内核

    • 内核进行模块化,删除不必要的功能.
    • 其他部件看成程序来实现
    • 各个模块采用消息传递进行通信

    优点:x

    • 内核变小
    • 拓展性高,移植性高

    可加载的内核模块

    • 在现在操作系统中很常用Linux Windws

    • 主内核知道如何加载模块和使得各个模块通信,不需要用消息机制

    • 动态加载内核模块

    进程

    概念

    引入
    • 并发执行,就是各个进程轮流使用CPU
    • 并行执行,需要多个CPU
    • 进程是执行中的程序实体,是操作系统进行资源分配和调度的基本单位
    进程的数据结构
    • 表示进程的数据结构是PCB , 所有的进程用双向链表进行连接,PCB保存进程上下文
    • 内核采用一个指针current,指向当前进程
    • CPU上下文切换实际上是PCB切换
    进程控制块
    • 进程状态(5个状态)与编号
    • 程序计数器,记录下一条指令的地址
    • 各种CPU寄存器
    • 各种堆栈信息,内存

    操作系统中的进程实际上就是进程控制块

    进程调度器

    目的
    • 提高CPU使用效率,在进程之间快速切换CPU
    • 进程调度器选择一个可用进程到CPU执行
    调度队列

    • 就绪队列:保存内存中就绪和等待的进程,用链表组织PCB

    • 设备队列:IO使用的队列

    调度程序分类

    根据提交程序的多少来进行划分,如果以下提交的程序太多,调度效率就很低

    • 长期调度程序: 提交的进程太多,先保存在磁盘中(缓冲池)
    • 短期调度程序:PCB较少,直接在内存中进行调度即可
    • 中期调度程序: 将一部分PCB移到磁盘中,减低PCB来调度,之后再把磁盘中的重新调入内存中

    第三种方式是交换swap

    上下文切换
    • 进程的上下文采用PCB表示
    • CPU的上下文切换就是PCB切换

    进程的运行

    那么如果我的程序想启动另一程序的执行但自己仍想继续运行的话，怎么办呢？那就是结合fork与exec的使用
    • 每一个进程可以创建一个新的进程,父进程,子进程,最终形成进程树
    • 每个进程用唯一标识符pid识别

    fork和exec函数的区别

    • fork创建一个新的进程, ,
      • 新进程复制父进程的地址空间
      • 这两个进程都继续执行fork后面的指令, 但是子进程fork返回值是0,父进程返回子进程的pid

    ​ 如果一个大程序在运行中，它的数据段和堆栈都很大，一次fork就要复制一次，那么fork的系统开销不是很大吗？其实UNIX自有其解决的办法，大家知道，一般CPU都是以”页”为单位来分配内存空间的，每一个页都是实际物理内存的一个映像，象INTEL的CPU，其一页在通常情况下是 4086字节大小，而无论是数据段还是堆栈段都是由许多”页”构成的，fork函数复制这两个段，只是”逻辑”上的，并非”物理”上的，也就是说，实际执行fork时，物理空间上两个进程的数据段和堆栈段都还是共享着的，当有一个进程写了某个数据时，这时两个进程之间的数据才有了区别，系统就将有区别的” 页”从物理上也分开。系统在空间上的开销就可以达到最小。

    • exec启动另外的进程以取代当前运行的进程。
      • 一个进程一旦调用exec类函数，它本身就”死亡”了，系统把代码段替换成新的程序的代码，废弃原有的数据段和堆栈段，并为新程序分配新的数据段与堆栈段，唯一留下的，就是进程号

    那么如果我的程序想启动另一程序的执行但自己仍想继续运行的话，怎么办呢？那就是结合fork与exec的使用

    进程终止
    • 系统调用exit删除进程自己,并释放资源

    系统调用wait()

    • 调用wait(),用来等待子进程的终止, 这个系统调用返回的是子进程的pid
    • 僵尸进程: 子进程终止,父进程还没掉用wait()进入的过度状态
    • 孤儿进程: 父进程没调用wait就终止了.此时处理方法: init()作为孤儿进程的父进程,并定时调用wait()

    进程间的通信

    两种基本方式
    • 共享内存 信号量来实现,信号量代表资源的使用情况
    • 消息传递
    客户机/服务器调用

    • 套接字 : 由一个五元组决定 ip和端口号确定 进行交换无结构的字节流,需要固定协议

    • 远程调用RPC : A客户机直接调用B客户机中的函数,最后得到B运行得到的值

    • 管道:进行进程间数据通信的一种简单方式

      • 普通管道 pipe() 单向通行,只有进程相互通信时,管道才存在,通信结束就不存在
      • 命名管道:双向的,不需要父子关系,同时多个线程也可以同他通信,不是实时性的.

      前者用于父进程和子进程间的通信，后者用于运行于同一台机器上的任意两个进程间的通信

    线程

    概念

    • 进程类似于从资源分配和调度的角度看
    • 线程是从代码执行的角度看，一个进程往往具有多个线程，

    区别：

    • 进程是操作系统进行资源分配和调度的基本单位

    • 线程是CPU使用的基本单元，线程是处理器调度的基本单位

    相关概念

    • 进程中的多个线程
      • 共用代码段，数据段等资源
      • 具有自己的寄存器、堆栈、线程id
    • 一个进程一半具有多个线程，进程就可以同时执行多个任务

    优势：

    • 一个多线程的程序响应时间快
    • 资源共享，上下文切换快
    • 进程的信息传递比较复杂

    多核编程

    • 多核：并行，依赖于多核同时运行多任务
    • 单核：并发，依赖于CPU上下文切换

    多线程模型

    用户级线程和内核级线程

    • 用户级进程不需要通过内核管理
    • 内核级线程需要经过操作系统来管理

    用户级线程和内核级线程是存 在对应关系的

    线程池

    在进程开始时，就创建一定数量的线程，并加入池中等待工作

    优点：

    • 使用已经存在的线程请求比等待创建一个快
    • 线程池限制创建线程的数量
    • 将执行任务和创建任务分离

    多线程问题

    fork()和exec()

    • exec()会覆盖替换进程中的所有线程

    • 需要一个轻量级进程（LWP）来处理用户线程和内核线程

    如果内核线程阻塞，LWP也会阻塞，导致用户线程也会阻塞
