    DBMS概念

    面向磁盘的DBMS

    面向磁盘的DBMS数据库管理系统总体运行规则是：数据按照块/页的方式保存在磁盘中，当需要操作数据的时候，数据库管理系统会将对于的页读入内存，同时在内存中维护一个缓冲池（Pooling Buffer） 来管理磁盘读入内存中的块。

    同时数据管理系统有一个执行引擎执行查询任务，主要在缓冲池中找到固定的开，然后缓冲池冲磁盘中读入内存，并返回内存指针到数据库引擎。

    DBMS和OS的区别

    DBMS主要用来管理数据远远超过内存的数据，实现类似于虚拟内存机制来管理内存与磁盘中间的读写。

    OS中用MMU实现内存映射，但是当出现page fault时会阻塞进程，但是我们希望事务不依赖于操作系统。

    但是DBMS可利用操作系统中的操作：

    • madivse告诉操作系统什么时候想要读取特定的块
    • mlock将每一块内存加锁，不被替换
    • msync将内存写入磁盘

    数据库文件存储

    页Page

    DBMS中的存储管理将数据转化为若干个页的集合，并检测一个文件中的页还有多少空间可以使用。

    每一个页都有一个唯一标识符（id）

    • 单文件系统中，这个标识符结束文件的偏移量
    • 多文件系统中，标识符需要被映射到文件路径和偏移量，只有只要才能找到这个也页

    DBMS中的页可以包含不同数据格式，数据混搭，同时页是自包含的，页本身的信息也写在页中，不需要额外的数据结构就可以知道该页的所有相关信息。

    DBMS的页一般是定长的，往往选择4kb

    Database Heap

    页的组织方式主要用堆文件(Heap File),可以让DBMS找到所需要的页在磁盘中对应位置

    堆文件中页的信息是用元组格=格式进行无需存储的。

    DBMS通过id找到磁盘中对应块的方法：

    • 链表查找很费时间，需要不断扫描
    • 页目录Page Directiry: DBMS维护一个专门的页用来记录每个页的数据页和空闲页的位置，和map类似1，用page id 直接查到对应页的位置。

    页存储格式

    每个页都包含了一个页头，头里包含了若干元数据，包括：

    • 页的大小
    • 校验和
    • 版本信息
    • 事务可见性
    • 自包含性

    头之后是页中的数据，一种naive approach是只是简单的记录一个页中存放了多少元组，每次有新的数据写入就在末尾添加，这样的做法会导致的问题是，如果数据被删除或者包含了不定长的字段，那么整个页的管理就会出现问题。常用的数据存储方式有两种，分别是slotted-pages和log-structured

    缓冲池

    缓冲池(Buffer Pool)就是数据库系统在内存中管理数据库页的一种方式，缓冲池实际上就是一个内存中的cache，并且由一系列固定大小的页组成一个数组，数组中的每一个条目(entry)叫做帧，在需要某个特定的页的时候，数据库系统会先从内存的缓冲池中找对应的页，如果找不到就去磁盘上将这个页读入内存中。

    数据库索引

    B+树索引

    B+树是一个自平衡的树结构，可以使得数据排好序，同时支持顺序查找，插入，删除，而且操作的复杂度都是：

    $$O(log\ n)$$

    B树（B-树）和B+树的区别

    • B+树的索引结点不保存数据，仅仅叶子结点保存值
    • B+树叶子结点都用指针从小到大链接起来
    • B+树叶子节点是有序的，索引结点也是从小到大排序的

    B+树优点

    • 仅仅叶子结点保存数据，索引节点不保存数据，使得一次性放进内存中的索引结点变多，IO次数变少，提高性能
    • 叶子结点形成一个有序链表，便于区间查询
    • B+树是完美平衡的，使得每次所用查询的时候复杂度都比较稳定
    • 于B+树在内部节点上不包含数据信息，因此在内存页中能够存放更多的key。 数据存放的更加紧密，具有更好的空间局部性

    B+树重复key

    • 解决方法：将数据的record ID也作为key的一部分，构造出特殊的key

    数据库中B+树的设计

    ==结点大小==

    • 一般数据叶子结点大小选取和页大小相同，以减小碎片

    ==变成Key时如何设计==

    • 索引结点保存Key的指针，但是这样效率可能会变低
    • 键值映射：最常用的方法还是用一个key-value对在一个单独字典中的索引来代替key，因为字典索引的规模比较小，因此有足够的空间在索引旁边放置key的前缀，并且可以允许索引的搜索和叶结点的扫描，避开指针的使用

    索引的并发控制

    ==锁Lock和闩Latch==

    锁和闩都可以用于并发的控制，但是两者其实存在一定的区别，具体的区别如下图所示：

    • 锁是一种高阶的逻辑原语，用来保护数据库中的内容不被其他事务所改变，事物会在它自己的整个持续过程中都持有锁，数据库可以在查询处理的过程中，将锁暴露给用户，而锁需要能够回滚更改。
    • 闩是一种低级的保护原语，用在具体的内部数据结构中，闩只在对数据结构的操作进行的过程中才持有，并且不需要回滚更改，闩有两种不同的形式：
      • READ：多个线程允许同时对一块内容进行读操作，一个线程可以在其他线程持有某个item的读闩的时候再申请一个读闩。
      • WRITE：只允许一个线程对某块内容进行写的操作，当一个线程持有某块内容的写闩时，其他任何线程不能获得对这块区域的读锁和写锁

    latch的实现往往是使用原子指令实现的（CAS compare-and-swap）

    B+树索引中的锁主要需要解决以下两个问题：

    • 多个线程尝试同时修改某个节点的内容
    • 一个线程在其他线程改变

    因此，B+树中锁的设计主要思路是进行锁的耦合(Coupling)，具体的原则是：

    • 先给父节点申请锁
    • 再给子节点申请锁
    • 如果父节点是安全的(不需要进行分割和合并)，那么就释放父节点的锁

    对于不同的操作，最基本的锁协议的处理方式是：

    • 搜索：从根节点开始搜索，重复“给子节点申请锁，给父节点释放锁”这一过程
    • 插入和删除：从根节点开始搜索，直到找到要访问的叶结点位置一直申请锁，然后检验叶结点是不是安全的，如果是安全的，那么就释放所有它的祖先的锁

    B+树的索引输入组织为文件

    • 将叶子节点设置为一个页的大小，实际上结点保存的这个页的指针
    • 通过叶子节点的指针可以对数据进行操作。叶子节点的指针是保存在索引文件中的，索引文件中的指针用来指向数据文件
    • 在内存中对这个页的修改会通过buffer实现对磁盘文件的修改

    不是innoDB的索引方式，innoDB是聚合索引，只有一个文件，不区分索引文件和数据文件

    为什么不用红黑树而是用B+树

    • 索引本身可能数据量很大，没办法完全保存在内存中，因此往往以索引文件的形式保存在磁盘中，因此B+树要比红黑树好
    • 红黑树是二叉树，深度很大，
    • 而且红黑树进行区间查询也没有B+树直接
    • B+树保存到磁盘中，是形成索引文件，需要自己定义文件格式，使得落盘满足单向增长的磁盘文件的需要
    • 磁盘地址只能单向增长，数据库要自己设计一个文件格式，让单向增长的地址能够适应增加方式复杂的B+树。详细内容可以参考 sqlite 源码
    • 红黑树的IO复杂度，IO次数要比B+树大，B+树对相邻数据的索引比较快

    B+树如何保存到索引文件中

    如何将树的结构保存成为字节流

    • 除了保存结点信息外，还需要保存父子关系
    • 实际上是树的序列化和反序列化

    如果每个节点是定长的比如一个结构体 就比较好办 从跟节点开始写入文件 再遍历每个字节点 只需要把指向子节点的指针 替换成文件中的偏移位置 再在该位置写入相应的字节点就行了

    问题的解决：

    • 将每一个B+树的结点（索引结点和叶子节点）都固定位应该页的大小
    • 仅仅需要将B+树中的指针替换成子节点在文件中偏移量即可，并在该位置写入子节点的数据
    • 初始化建立B+树时，根据偏移量递归建树即可

    2、缓冲区的设计

    如果每一次访问记录都要进行打开文件，关闭文件的操作速度会很慢。我就设计了一个缓冲区用来提高读取速度。

    我们将磁盘文件每4KB个字节作为一个页，缓冲区有1000个页，每一次读取都会现在缓冲区里面找数据，如果miss了，会去磁盘文件读取指定记录4KiB的数据到缓冲区，如果缓冲区满了就进行LRU页面替换和clock。这样在去缓冲区读取就hit击中了，下次再读取一个记录就可以直接再缓冲区里面找了。

    总体来说就是我通过buffer类向外通过接口，通过堆buffer的读写实际上相当于对底层文件的读写。

    MVCC是什么 同时用到锁和log (回滚指针指向undo log)

    1、MVCC是什么

    InnoDB的多版本并发控制，优势是读不加锁，读写不冲突，读写分离，并发控制优势很大

    2、解决什么问题

    • 脏读
    • 不可重复读
    • 幻读
    • 不能解决丢失更新的问题

    3、原理是什么

    MVCC实现由： 3个隐式字段，undo日志 ，Read View 三部分组成

    1、mvcc会在每一行后面加上隐藏字段，表示创建时间和删除时间，创建时间实际上是版本号，事务的id，回滚指针

    2、每一次执行一个新的事务版本号会增加

    3、undo log保证事务原子性

    4、读视图是用来保证可见性的，

    • 可重复读中，事务开始时创建一个视图，后面的操作都用一个视图，读到版本号等于当前版本号或者早于当前版本号的数据
    • 读提交时，每个语句执行都会创造一个视图，每次都是读到版本号最新的数据行

    4、使用的隔离级别

    仅仅在可重复读和提交读的隔离性下工作 都是通过版本号来操作的

    提交读 就是读取最新版本号的数据行

    可重复读： 查找版本号早于或者等于当前版本号的数据行

    在可重复读的基础上没有完全解决幻读的问题，还需要 RR上加上间隙锁

    所以引发了 当前读 和 快照读

    当前读 会加锁，保证读取的最新的版本

    快照读 性能考虑，但是读到的不一定是最新的 selete不加锁就是

    ==中断和异常的区别==

    • 中断也叫外中断，来自处理器外部的中断信号，根据优先级部分中断是可以屏蔽的
    • 异常也叫内中断，来只处理器内部的中断信号，不能被屏蔽

    ==为什么中断处理过程不能被阻塞，但是异常处理可以阻塞？==

    中断处理程序启动后，不响应其他中断请求，因为要求中断被快速响应。

    异常处理可以在处理程序中设置阻塞。

    ==中断处理和异常处理的具体区别==

    对于中断处理程序，中断控制器发现中断源（外部设备、时钟部件或其它计算机发出）并产生中断，当前指令执行结束后， CPU 会检查中断寄存器（中断控制器的一部分）是否有中断事件发生，如果发现中断信号将暂停执行当前程序，转向执行内核的中断处理程序，如果没有发现中断信号，则继续执行当前程序。

    对于异常处理程序，没有中断源一说，不使用中断控制器，是处理器在执行指令时本身发生的异常，指令控制逻辑和实现线路一旦发现异常变转向内核的异常处理程序。

    ==中断和异常都会使得操作系统转向内核态==

    ==处理器从用户态转到内核态的三种方式==

    • 用户进程执行陷阱指令（例如系统调用）转到内核态
    • 导致异常，进入内核异常处理程序中执行
    • 接收到中断，转到内核

    ==向操作系统转递参数的三种方式==

    • 直接用寄存器传递
    • 寄存器传递参数地址
    • 保存堆栈，os弹出堆栈

    ==用户态和内核态的区别==

    • 内核态:完全使用硬件，可以执行任何CPU指令，获得任何内存地址

    • 用户态：代码无法直接访问硬件或参考内存。只能通过系统调用访问硬件

    • 保护机制

    ==描述内核在进程之间切换上下文所采取的操作。==

    • 保存当前状态 PCB
    • 进程调度器选择下一个进程 current指针指向下一个PCB
    • 切换到下一个进程运行 PCB

    ==多线程共享进程中哪些东西，哪些不是共享的==

    • heap memory and global variables. 代码段和堆内存共享
    • **register ** and stack 寄存器和栈，线程ID是独立的

    698. 划分为k个相等的子集

    不会做没思路，但是这道题很重要的样子

    状态压缩

    给定一个整数数组 nums 和一个正整数 k，找出是否有可能把这个数组分成 k 个非空子集，其总和都相等。

    示例一：

    思路：
    为看到数据范围为16，就可以想到用状态压缩来做。
    首先，容易想到的是，如果所有数之和不能被k整数，那么一定是不可以的。并且，每个组的数之和为$target = sum(nums) / k$

    其次考虑状态压缩，如果每个数都能用0表示为不选择，1表示为选择，假设数组的长度为n，那么一共有$2 ^ n$种状态

    下面考虑dp，使用dp[i]表示第i种状态能否分为若干个之和为target的数组，那么最终的答案是$dp[2^n-1]$,也就是将数组中所有数都选上，这个数组能够分为若干个和为target数之和。
    这里使用了一个技巧，也就是$dp[i]$表示的是当前状态求和之后%target的余数，除了dp[0]以外，其余状态%target等于的0话，就表明一定存在某种划分，所以只需要判断最后一个状态是否为0即可。
    那么状态转移为

    $$dp[mask|(1<<i)] = (dp[mask]+nums[i]) % target$$

    注意$dp[mask|(1<<i)]$表示的是给当前状态mask，的第i为变成1，所以我们要保证这一位之前不是1，当然我们要保证dp[mask] + nums[i]不能是一个大于target的数,因为nums[i]不能再拆了，有了这两个条件就可以状态转移了.
