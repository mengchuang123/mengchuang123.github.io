Prim算法

  • 选取一个顶点作为已选择的顶点，加入已选结点集合

  • 选出未选结点集合中选出一个和已选结点集合中的距离最小的结点

  • 该结点加入已选结点集合

证明：

用归纳法（反证法）证明形成的MST

  • 假设一个最短，但是不是最优的结点，和另外一个最优但是不是最短的边，
  • 两个不可能同时存在，必然只能选择一个
  • 明显最短的那个就是最优的。

效率：

  • 权重矩阵表示： $O(|V|^2)$
  • 邻接链表表示： $(|V|-1+|E|)O(log|V|)=O(|E|log|V|)$

如何判断最小生成树是不是唯一的呢

  • 如果全部的边都是不一样的，那么生成树就是唯一的。
  • 无向图中存在相同权值的边是最小生成树不唯一的必要条件（但不是充分条件）

判定最小生成树唯一的算法思路

  • 求次小生成树，判断次小生成树的大小是否等于最小生成树

次小生成树

  • 求第二小生成树的时候，一般采用的是求出最小生成树后，
  • 依次删除最小生成树上的每一条边，然后生成n-1次最小生成树，
  • 记录下这个过程中的最小的生成树的值，那么这个就是第二小生成树了。

1.HTTP特性

  • HTTP建立在TCP/IP上的应用层协议，默认端口为80
  • HTTP协议是无状态，无连接的，通信双方无需建立连接；
  • HTTP协议是一种请求-应答式的协议——客户端发送一个请求，服务器返回该请求的应答，所有的请求与应答都是HTTP包

HTTPS在HTTP上面加上SSL

1.HTTP和HTTPS的区别

  • HTTP 协议以明文方式发送内容，不提供任何方式的数据加密
  • HTTPS 经由 HTTP 进行通信，但利用 SSL/TLS 来加密数据包。
  • HTTP 明文传输，数据都是未加密的，安全性较差，HTTPS（SSL+HTTP） 数据传输过程是加密的，安全性较好
  • 使用 HTTPS 协议需要到 CA（Certificate Authority，数字证书认证机构） 申请证书，一般免费证书较少，因而需要一定费用。
  • HTTP 页面响应速度比 HTTPS 快，主要是因为 HTTP 使用 TCP 三次握手建立连接，客户端和服务器需要交换 3 个包，而 HTTPS除了 TCP 的三个包，还要加上 ssl 握手需要的 9 个包，所以一共是 12 个包。
  • http 和 https 使用的是完全不同的连接方式，用的端口也不一样，前者是 80，后者是 443。
  • HTTPS 其实就是建构在 SSL/TLS 之上的 HTTP 协议，所以，要比较 HTTPS 比 HTTP 要更耗费服务器资源

2.HTTPS工作流程

HTTPS 默认工作在 TCP 协议443端口

  • TCP 三次同步握手
  • HTTPS处理TCP创建连接后，需要加上SSL握手的9个包

cache原理

从以下五个方面进行介绍

  • cache的目的

  • cache的结构

  • cache的访问

  • cache的缺失处理

  • cache的写操作

1.cache的目的

  • 提高速度
  • cpu运行速度远远快于数据传输速度

2.cache结构

我们将内存地址划分为三个部分：

  • Tag
  • index
  • offset(byte offset)

cache的基本数据单元是块(block),长度为块数，用index来查找，块的宽度是byte offset,为了判断当前块是不是寄存器地址中的值，还需要再cache前面加上tag位，同时加上valid一位。

3.cache的写操作

  • 写直达：更新cache的同时，更新内存

  • 写回

    • 更新cache中的word
    • 允许先不更新主内数据
    • 在cache前面添加一个dirty位，来代表cache和主存内容的不一致性，当块被替换的时候，需要更新主存
    • OS通过IO向外部读写数据的时候，会更新cache中的内容到主存中，也就是flush

4.cache缺失处理

三种miss

  • 强制性失效CPU第一次访问相应cache块，cache中肯定没有该cache块，引起的失效叫做强制性失效。这是不可避免的。

    • (无限容量 全相连)

  • 容量失效：有限的cache容量导致cache放不下而替换出cache块，被替换出去的cache块再被访问，引起的失效叫做容量失效。

    • (有限容量 全相连)

  • 冲突失效：在直接相联或组相联的cache中，不同的cache块由于index相同相互替换，引起的失效叫做冲突失效。

冲突缺失：如果高速缓存完全相关联且采用LRU替换规则，则不会发生冲突缺失。

容量缺失：即使采用全关联和最完美的替换策略也会发生容量缺失。

例如有一个大小为4的直接映射高速缓存。访问序列如下：0,1,2,3,4,1,2,3,0,4,0

0(compulsory miss), 1(compulsory miss), 2(compulsory miss), 3(compulsory miss), 4(compulsory miss), 1(hit), 2(hit), 3(hit), 0(capacity miss), 4(capacity miss), 0(conflict miss)

倒数第二个0是容量丢失，因为即使该缓存与LRU缓存完全关联，它仍然会导致丢失，因为在最后0之前访问了4,1,2,3。但是最后一个0是一个冲突丢失，因为在完全关联的缓存中，后4个将替换缓存中的1而不是0

cache如何进行优化

考虑以下因素

  • 块大小：随着块大小增大，缺失率会变小（空间局部性）在变大（块数量减小影响空间局部性）

  • 直接变大cache

1.通过更加灵活放置块，来减少cache缺失

全相连

  • 直接取消index，而是直接比较tag，需要添加大量的比较器并行化处理

组相连

  • n路组相连：每一个index都对应n个块，在查找是不知道是哪个块，因此需要对每个块按照tag进行对比，找到哪个块以后，在按照byte offset来进行查找。
  • index的位数： 块数/n 来确定
  • 组相连：index用来查找哪个组，tag用来确定哪个块，offset是块中被请求的数据地址

缺点/优势

  • 提高相连度减低复杂度
  • 缺点是增加命中时间

LRU:最近最少使用

1.1Prim

  • 选取一个顶点作为已选择的顶点，加入已选结点集合

  • 选出未选结点集合中选出一个和已选结点集合中的距离最小的结点

  • 该结点加入已选结点集合

证明：

用归纳法（反证法）证明形成的MST

  • 假设一个最短，但是不是最优的结点，和另外一个最优但是不是最短的边，
  • 两个不可能同时存在，必然只能选择一个
  • 明显最短的那个就是最优的。

效率：

  • 权重矩阵表示： $O(|V|^2)$
  • 邻接链表表示： $(|V|-1+|E|)O(log|V|)=O(|E|log|V|)$
