1、介绍

尽管有很好的灵活性，但这种方法通常不能有效地利用 GPU 资源。
例如，对于 GNMT [42] 等常见翻译模型和 RHN [47] 等语言模型，大多数计算资源都处于空闲状态。这些训练算法包括许多 RNN 模块 [28]，例如 LSTM [12] 和 GRU [8] 网络，它们对 GPU 的数据并行度很小，导致 GPU 资源的利用不足。
作为一种不同的方法，空间共享可以提供比时间共享更好的吞吐量，只要单个训练作业不会完全饱和 GPU 计算资源 [44]。然而，应用空间共享的一个限制是并发作业的工作集大小，它随着工作协同定位而显着增长。如果工作集不适合 GPU 内存，则系统必须终止作业或将 GPU 内存交换到主机，这会掩盖空间共享的性能优势。因此，要使空间共享更广泛地适用，必须减少协同训练作业的内存占用。

我们观察到共享在同地训练作业期间生成的中间数据显着减少了总内存占用。训练是一个高度迭代的过程，首先为每批输入数据按顺序导航层（前向传递），然后以相反的顺序（后向传递）导航相同的层。在训练过程中，来自称为特征图的模型层的中间输出支配着内存占用[18,32]。特征图在前向传递期间在每一层中生成，然后在后向传递中消耗以更新层。由于常规的双向执行，单个训练作业中的内存消耗通常呈现出循环模式——内存消耗在前向传递中逐渐增加，然后在后向传递中减少。因此，节省内存消耗的一个简单而有效的策略是创建一个大型 GPU 内存池，并为并发训练作业弹性共享内存池。为了增加共享机会，需要协调并发训练作业以使它们运行不同的传递，例如，作业 A 的前向传递（增加其工作集）和作业 B 的反向传递（减少其工作集）。这个方法导致一个作业的内存分配与另一个作​​业的内存释放同时发生，从而有效地减少了系统范围的内存占用。

尽管共享的想法是合理的，但今天的 DNN 框架在 GPU 上执行训练的方式提出了重大挑战。当前的框架主要是为单独的训练案例设计的。根据数据流依赖性，它们提前分配每个 DNN 内核计算所需的内存，并向 GPU 流发出尽可能多的内核，即每个作业的 GPU 计算工作队列，以使 GPU 的内部计算资源饱和。这导致 GPU 计算与 CPU 处理异步且并行。因此，平台不知道 GPU 计算的进度以及 GPU 何时确实消耗了内存。如果不正确处理异步，共享内存不能保证正确性，例如防止其他训练作业在等待内核的共享未开发内存中损坏内存。

在本文中，我们提出了 Zico，这是一个 DNN 平台，可以为并发训练提供高效的内存共享。 Zico 对广泛使用的 DNN 平台 TensorFlow 进行了改造，以最大限度地提高并发训练的整体吞吐量。 Zico 的目标是找到并发训练的最佳协调执行，以充分利用 GPU 计算和内存资源。为实现该目标，
(i) Zico 准确监控训练作业的计算进度。基于此，Zico 为 DNN 内核分配和取消分配内存，从而更接近 GPU 的视图通知内存使用模式。
(ii) Zico 包含运行时信息（例如，迭代时间、内存使用模式和 GPU 内存限制）并执行称为 scrooge 调度程序的作业调度程序，以有效地引导并发作业以利用共享内存池。
(iii) Zico 有效地将整个 GPU 空间组织为一个弹性共享内存池，以支持 scrooge 调度程序。

为了检测异步内核的计算进度，Zico 利用称为 CUDA 事件的特定内核，通知 GPU 内核的进度。 Zico 使用 CUDA 事件来识别 GPU 内核使用的内存的分配和释放时间。基于这些信息，Zico 执行我们新颖的 scrooge 调度程序来预测迭代边界处并发训练的内存消耗趋势，并在每次迭代中引入最小停顿时间。然而，协同调度作业的内存使用趋势会根据它们在使用 GPU 计算单元时相互干扰的方式而有所不同。为了应用它们的动态行为，scrooge 调度程序根据运行时收集的反馈细化决策

Zico 将内存池组织为称为区域的块的集合，并根据数据特征分离它们的用途。
DNN 训练生成几种类型的数据作为张量，主要分类为具有高发生率的短暂张量和长寿命的张量，例如构成最多内存占用的特征图。通过按类型分离区域，Zico 确保存储有特征图的内存不会干扰其他瞬态数据，同时使它们的
需求遵循训练迭代的循环模式。这种设计选择使我们的调度决策可以在很少中断的情况下应用，而不会失去共享机会。

对于高内存占用，Zico 分别比传统的空间共享和时间共享方法快 8.3 倍和 1.6 倍，尤其是在同时训练不同模型时。此外，对于低内存占用，不需要停止并发训练，Zico 的行为类似于传统的空间共享，并且比时间共享快 1.6 倍。总体而言，Zico 实现了加速，无论并发训练是基于相同还是不同的模型。

由于并非所有用例都需要训练作业之间的强隔离，例如由单个用户驱动的超参数调整 [26]，因此最近的工作支持在单个进程域中类似于 MPS 的空间 GPU 共享 [44]。无论保护级别如何，在 GPU 内实现空间共享的底层机制都非常相似，即使不相同 - 所产生的性能也是如此。

三个挑战

内存共享的挑战 GPU 的硬件资源有限，需要高效使用。由于 GPU 的计算和内存资源被共享以运行并发训练，限制了 pertraining 资源容量，因此彻底了解 DNN 框架中的当前实践并发现空间 GPU 训练的挑战至关重要。在本节中，我们将讨论 Zico 需要解决的三个主要挑战。

1、内存膨胀

特征图导致峰值内存消耗大大高于每次迭代中模型大小对应的最小值。我们将此问题称为内存膨胀。

传统的空间 GPU 共享机制容易受到内存膨胀的影响。像 TensorFlow 这样的 DNN 框架并不倾向于设计用于与内部内存管理器共享内存，维护本地内存池以进行单次训练。因此，不会发生对 GPU 的内存释放。
这导致并发训练的内存使用量达到峰值，所有这些都加起来并因此使 GPU 内存饱和，即使在单个训练的内存占用量不大的情况下也是如此。
为了说明这一点，让我们以两个相同模型的并发训练为例，每个模型都需要超过一半的 GPU 内存。在这种情况下，两个本地内存池的内存需求超过了 GPU 内存容量，开始利用 CPU 端内存作为交换空间。为了促进这一点，包括 V100 在内的最新 NVIDIA GPU 提供了一种称为统一虚拟内存 (UVM) 的功能，该功能对 DNN 平台是透明的。
我们发现，尽管灵活性很大，但使用 UVM 进行 DNN 训练目前成本高昂且严重影响整体性能。为了确认效果，当训练作业占用 70% GPU 内存时，我们比较了使用 NVIDIA V100 的 ResNet50 单独训练与并发训练之间的吞吐量。为了公平起见，我们将单个训练作业配置为使用 MPS 设置的 50% GPU 资源。
由于 GPU 内存超额使用，并发训练的吞吐量急剧下降（即慢 8 倍）。因此，我们应该降低并发训练期间 GPU 内存被用完的风险。

设计

scrooge 调度器
在观察内存使用情况时，Zico 调度并发作业以在训练中实现最佳性能，同时将总内存消耗置于预定义的预算内，不超过 GPU 内存容量。

出给定的内存预算。一种节省内存消耗的简单方法是协调训练作业，使前向执行（即内存分配阶段）与后向执行（即内存释放阶段）尽可能重叠。

为了给内存共享提供具体的指导，我们设计了 scrooge 调度器，它促进了一个目标函数，帮助预测未来系统范围的内存需求。在开始新的迭代之前，调度程序会考虑正在使用的内存区域的寿命，这些内存区域是 Zico 中的可共享内存单元。然后，调度器预测是否立即允许迭代消耗的内存是否小于内存预算。如果这几乎是不可能的，那么调度程序会估计在新迭代上应用的最小停顿时间，以便在有更多内存可用时安全地完成稍后发出的前向传递中的内存分配。为了在调度中做出正确的决定，必须保持精确的区域寿命。为了实现这一目标，Zico 根据在运行时从先前迭代中收集的反馈迭代地优化区域生命周期。
